RPC는 `Remote Procedure Call`의 약자이다.

풀어쓰면 “원격 프로시저 호출” → 클라이언트에서 원격에 존재하는 프로시저를 네트워크 통신을 통해 로컬에서 사용하는 것처럼 호출하는 것을 말한다. 이때 클라이언트에서는 직접적으로 프로시저를 호출하지 않고 중간에 stub이라는 객체를 통해 원격 서버로 호출을 전달한다.

RPC는 말그대로 프로시저를 호출하기 때문에 정해진 주소로 호출하는 REST API와는 형태가 다르다.

```jsx
// REST 방식
const res = await fetch('/api/users/1');
const user = await res.json();

// RPC 방식
const user = await api.users.getById({ id: 1 });
```

RPC의 경우 하부 네트워크 프로토콜 구조에 신경쓰지 않고 고유 프로세스 기능 개발에 집중할 수 있고 네트워크 통신의 세부사항을 숨길 수 있다는 장점이 있지만, 네트워크에 크게 의존적이라 원격 호출을 했을 때 반환 시간을 보장하지 않는다는 단점이 있다.

---

### 쉽게 설명하자면

RPC는 서버에 있는 함수를 네트워크를 통해 마치 내 코드안의 함수처럼 호출하는 기술이다.

즉, 개발자는 `await api.users.getById({ id: 1 });` 처럼 단순히 함수를 부르지만, 사실 이 호출은 네트워크를 통해 원격 서버로 전달되고, 서버에서 실행된 결과가 다시 클라이언트로 반환되는 구조이다.

이 방식은 네트워크 요청(fetch, HTTP 등)을 직접 다루지 않아도 돼서 코드가 단순하고 비즈니스 로직 개발에 집중할 수 있다는 장점이 있다.

하지만 결국 네트워크 통신에 의존하므로, 서버 상태나 네트워크 지연에 따라 응답 속도가 불안정하거나 실패할 수 있다는 단점이 있다.
